Microfrontend Architecture

Runtime Module Composition

A browser-native strategy for composing independently deployed frontend slices with ESM, import maps, and dynamic module loading.

Overview

Runtime Module Composition turns the browser into the composition runtime. A host shell loads a shared import map, resolves the active route to a module specifier, and dynamically imports the frontend slice responsible for that part of the product.

Ferry RSVP uses this pattern to keep feature slices independently built and deployed while preserving one coherent user experience.

Host Shell

Owns the document, root providers, locale-aware routing, navigation, persistent layout, loading states, and error boundaries.

Import Map

Centralizes framework versions, internal module prefixes, shared utilities, UI packages, and static asset origins.

Feature Slices

Build as ESM, publish stable module entry points, externalize shared runtime dependencies, and own product-specific UI.

Implementation Model

  1. The browser requests the host HTML.
  2. The host loads the import map before application modules run.
  3. The shell boots global providers and resolves the current route.
  4. The dynamic loader imports the slice module for that route.
  5. The browser resolves shared dependencies through the import map.
  6. The slice renders inside the host frame.

Operational Rules

  • Runtime dependencies belong in the import map.
  • Slices build and publish ESM outputs.
  • The route resolver is part of the runtime contract.
  • Shared dependency upgrades require cross-slice smoke testing.
  • Rollback should be possible by restoring a previous import map.
